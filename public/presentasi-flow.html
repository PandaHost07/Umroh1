Dokumen presentasi — internal / stakeholder

Ada Tour Travel

Sistem manajemen perjalanan umrah berbasis web: peran jamaah, admin operasional, dan admin keuangan — dengan penjelasan alur bisnis, autentikasi, model data, dan letak kode di proyek Next.js.

Cara pakai dokumen ini: gunakan menu di panel kiri (desktop) atau tombol « Menu isi » (HP/tablet) untuk pindah bab secara instan. Diagram membutuhkan internet (pustaka Mermaid dari CDN). Untuk presentasi layar penuh, disarankan browser Chrome atau Edge.

1. Ringkasan sistem

Bab ini menjawab: apa yang dibangun dan siapa penggunanya, sebelum masuk ke alur teknis.

Tujuan: mendigitalkan proses mulai dari promosi paket, pendaftaran jamaah, pembayaran bertahap, pengumpulan dokumen, hingga pelaporan keuangan.

Arsitektur singkat: satu aplikasi Next.js (tampilan halaman + API di folder app/api), database PostgreSQL diakses lewat Prisma. Di lingkungan cloud, koneksi bisa melalui Prisma Accelerate (URL database berawalan prisma+postgres://) agar query tetap cepat dari serverless.

Modul utama

  • Publik — beranda, paket, galeri, tentang kami, testimoni.
  • Autentikasi — registrasi jamaah, login (email/password), sesi JWT + catatan di tabel Session.
  • Jamaah — transaksi, pembayaran & bukti, dokumen, profil, keberangkatan, testimoni.
  • Admin operasional — master paket, hotel, penerbangan; data jamaah; dokumen & perlengkapan; pengumuman; mitra.
  • Admin keuangan — daftar pembayaran, verifikasi, laporan.

2. Peran pengguna & struktur URL

Satu akun punya satu role di tabel Akun. URL dashboard mengikuti role itu persis (huruf besar/kecil mengikuti enum Prisma), agar middleware bisa memblokir akses silang.

Role di database (enum Role) harus konsisten dengan prefix rute. Middleware memaksa pengguna hanya mengakses area sesuai role.

Role (Prisma) URL utama Deskripsi
jamaah /jamaah Pengguna akhir setelah daftar/login
ADMIN_OPERASIONAL /ADMIN_OPERASIONAL Operasional: paket, jamaah, dokumen, pengumuman, dll.
ADMIN_KEUANGAN /ADMIN_KEUANGAN Keuangan: transaksi, verifikasi pembayaran, laporan
Catatan presentasi: jika pengguna sudah login lalu membuka /login, sistem mengarahkan ke /{role}. Akses ke area role lain akan diarahkan kembali ke area sendiri.

3. Alur middleware & proteksi rute (detail)

Middleware berjalan sebelum halaman dimuat untuk path seperti /jamaah/... dan /ADMIN_*. Ia tidak menggantikan validasi di API; API tetap harus memastikan user hanya mengubah data yang diizinkan.

Setiap permintaan ke halaman yang dilindungi dicek: ada JWT tidak, sessionToken masih valid di database tidak, dan apakah path cocok dengan role.

4. Alur login & sesi (sequence diagram)

Login memakai email + password (Credentials). Password disimpan sebagai hash bcrypt di Akun.password. Setelah sukses, ada dua jejak sesi: cookie NextAuth (JWT) dan satu baris di tabel Session untuk dicek middleware.

NextAuth memakai strategi JWT. Setelah password benar, server membuat baris baru di tabel Session dan menyematkan sessionToken ke dalam JWT.

sequenceDiagram
  autonumber
  participant U as Pengguna / Browser
  participant NA as NextAuth
  participant DB as PostgreSQL via Prisma

  U->>NA: Kirim email + password (Credentials)
  NA->>DB: findUnique Akun by email
  alt Akun tidak ada
    DB-->>NA: null
    NA-->>U: Error kredensial
  else Akun ada
    DB-->>NA: Akun + hash password
    NA->>NA: bcrypt.compare(password, hash)
    alt Password salah
      NA-->>U: Error kredensial
    else Password benar
      NA->>NA: Generate sessionToken acak
      NA->>DB: INSERT Session (sessionToken, userId=email, expires)
      NA-->>U: JWT berisi sessionToken, role, nama, email, ...
    end
  end

Langkah setelah login

  1. Browser menyimpan sesi NextAuth (cookie).
  2. Setiap navigasi ke area terlindungi, middleware membaca JWT → validasi Session di DB.
  3. Role efektif diambil dari relasi Session → Akun, sehingga perubahan role di DB dapat dipakai untuk kontrol akses.

9. Arsitektur lapisan (website + coding)

Client = browser (React). Server = Node saat Next merender halaman atau menjalankan route.js. Database tidak pernah dihubungi langsung dari browser; selalu lewat server (Prisma).

Alur teknis dari sisi browser hingga database — berguna untuk menjelaskan di mana kode berjalan (client vs server).

10. Dua pola request yang sering dipakai

Proyek ini banyak memakai komponen bertanda "use client" yang memanggil fetch('/api/...') setelah halaman tampil. Pola alternatif adalah mengambil data di Server Component tanpa fetch ke API sendiri — keduanya valid di Next.js.

flowchart LR
  subgraph A[Halaman dengan data awal]
    U1[Pengguna buka URL] --> S1[Server render page]
    S1 --> S2[Bisa query prisma di Server Component]
    S2 --> S3[HTML dikirim ke browser]
  end
  subgraph B[Aksi dari tombol / form]
    U2[Klik submit / pakai useEffect] --> F1[fetch ke /api/...]
    F1 --> R1[route.js validasi + prisma]
    R1 --> F2[JSON response]
    F2 --> F3[UI update state / reload]
  end

Praktik di proyek ini: banyak halaman admin/jamaah memuat data lewat fetch dari client ke /api/system/... atau /api/jamaah/.... NextAuth menangani /api/auth/* secara khusus.

  if (paket.status !== "AKTIF") { /* 400 */ }
  const used = await hitungKuotaTersedia(prisma, paketId);

13. Konvensi coding & hal yang sering salah

Bagian ini merangkum kesalahan umum (salah URL role, bocor env ke client, seed usang) agar tim menghindari debugging berulang.

  • Import alias — @/lib/prisma mengacu ke root proyek (lihat jsconfig.json paths).
  • Role vs URL — nilai enum Prisma harus sama dengan segment URL (ADMIN_KEUANGAN huruf besar, jamaah huruf kecil).
  • Session ganda — tiap login menambah baris Session; pertimbangkan pembersihan untuk produksi.
  • prisma/seed.js — belum selaras skema terbaru; untuk admin gunakan npm run seed:file-data.
  • Environment — variabel tanpa prefix hanya ada di server; yang berawalan NEXT_PUBLIC_ terbuka ke browser.

Variabel lingkungan vs lapisan

14. Stack teknologi & menjalankan proyek

Perintah di kotak kuning di bawah adalah langkah minimal untuk mencoba aplikasi di komputer lokal setelah kode di-clone.

  • Next.js 16 + React 19 — UI dan server components / route handlers.
  • Prisma + PostgreSQL — model data; opsional Accelerate untuk URL prisma+postgres://.
  • NextAuth — JWT + tabel Session.
  • Firebase Admin / Vercel Blob — penyimpanan file.
  • Nodemailer — email SMTP.
  • Tailwind + Flowbite — tampilan.
Untuk menjalankan proyek: npm install, atur .env, npx prisma generate, npm run dev. Seed akun admin contoh: npm run seed:file-data (lihat file-data.json).
